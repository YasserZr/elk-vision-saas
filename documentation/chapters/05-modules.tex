\chapter{Developed Modules}

\section{Backend Modules}

\subsection{Users Module (\texttt{app.users})}

The users module handles authentication and user management:

\begin{itemize}
    \item \textbf{Views}: Registration, login, logout, profile management
    \item \textbf{Serializers}: User data validation and transformation
    \item \textbf{Authentication}: Token-based authentication with DRF
    \item \textbf{Permissions}: Custom permission classes for access control
\end{itemize}

\textbf{Key Endpoints}:
\begin{itemize}
    \item \code{POST /api/auth/register/} - Create new user account
    \item \code{POST /api/auth/login/} - Obtain authentication token
    \item \code{POST /api/auth/logout/} - Invalidate token
    \item \code{GET /api/auth/user/} - Get current user information
\end{itemize}

\subsection{Logs Module (\texttt{app.logs})}

The logs module is the core of the application, handling log ingestion, search, and metadata:

\textbf{Components}:
\begin{itemize}
    \item \textbf{views.py}: REST endpoints for log CRUD operations
    \item \textbf{views\_search\_history.py}: Search history tracking
    \item \textbf{views\_metadata.py}: Log metadata management
    \item \textbf{models\_mongo.py}: MongoDB document models
    \item \textbf{parsers.py}: Log file parsers (JSON, CSV, TXT)
    \item \textbf{logstash\_forwarder.py}: TCP/UDP log forwarding
    \item \textbf{tasks.py}: Celery async processing tasks
    \item \textbf{serializers.py}: Data validation and formatting
\end{itemize}

\textbf{Key Endpoints}:
\begin{itemize}
    \item \code{GET /api/logs/} - List paginated logs
    \item \code{POST /api/logs/upload/} - Upload log file
    \item \code{GET /api/logs/search/} - Search with filters
    \item \code{GET /api/logs/stats/} - Get statistics
    \item \code{GET /api/logs/search-history/} - Search history
\end{itemize}

\subsection{Dashboards Module (\texttt{app.dashboards})}

Manages user dashboard configurations:

\begin{itemize}
    \item Create and customize dashboard layouts
    \item Save widget configurations
    \item Share dashboards between users
\end{itemize}

\subsection{Alerts Module (\texttt{app.alerts})}

Implements the alerting system:

\begin{itemize}
    \item Define alert rules with conditions
    \item Set thresholds and time windows
    \item Configure notification methods
    \item Track alert history
\end{itemize}

\subsection{Health Module (\texttt{app.health})}

Provides health check endpoints for monitoring:

\begin{itemize}
    \item \code{GET /api/health/} - Overall system health
    \item Database connectivity checks
    \item Service availability verification
\end{itemize}

\subsection{API Module (\texttt{api})}

Core API infrastructure:

\begin{itemize}
    \item \textbf{consumers.py}: WebSocket handlers for real-time streaming
    \item \textbf{realtime.py}: Redis Pub/Sub to WebSocket bridge
    \item \textbf{routing.py}: WebSocket URL configuration
    \item \textbf{management commands}: Custom Django commands
\end{itemize}

\section{Background Services}

\subsection{Celery Worker}

Executes asynchronous tasks:
\begin{itemize}
    \item Log file processing and parsing
    \item Elasticsearch index operations
    \item Alert condition evaluation
    \item Report generation
\end{itemize}

\subsection{Celery Beat}

Schedules periodic tasks:
\begin{itemize}
    \item Index cleanup and optimization
    \item Statistics aggregation
    \item Health check monitoring
\end{itemize}

\subsection{Real-Time Listener}

Custom Django management command that:
\begin{itemize}
    \item Subscribes to Redis \code{logs:realtime} channel
    \item Parses incoming log messages
    \item Broadcasts to Django Channels groups
    \item Maintains connection health
\end{itemize}
